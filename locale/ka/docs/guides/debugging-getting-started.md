# გამართვის სახელმძღვანელო

ეს სახელმძღვანელო დაგეხმარებათ მიიღოთ დაიწყო გამართვის თქვენი Node.js პროგრამები და სკრიპტები.

## საშუალებას ინსპექტორი

როდესაც დაიწყო **--შეამოწმოს** შეცვლა, Node.js პროცესი უსმენს მეშვეობით WebSockets
დიაგნოსტიკური ბრძანებები, როგორც განსაზღვრული [ინსპექტორი ოქმი][],
სტანდარტულად ერთი მასპინძელი და პორტი 127.0.0.1:9229. თითოეული პროცესი, ასევე ენიჭება
უნიკალური [UUID][] (მაგალითად, `0f2c936f-b1cd-4ac9-aab3-f63b0f33d55e`).

ინსპექტორი კლიენტებს უნდა იცოდეს და მიუთითოთ მასპინძელი მისამართი, პორტი, და UUID დაკავშირება
რომ WebSocket ინტერფეისი. სრული URL არის
`ws://127.0.0.1:9229/0f2c936f-b1cd-4ac9-aab3-f63b0f33d55e`, რა თქმა უნდა, დამოკიდებული
ფაქტობრივი მასპინძელი და პორტი და სწორი UUID, რომ მაგალითად.

ინსპექტორი ასევე მოყვება HTTP endpoint ემსახურება მეტადატის შესახებ debuggee,
მათ შორის, მისი WebSocket URL, UUID, და Chrome DevTools URL. მიიღოს ამ მეტადატის
გაგზავნით HTTP მოთხოვნით `http://[მასპინძელი:port]/json/სია". ეს ბრუნდება
JSON ობიექტი, როგორიცაა შემდეგ; გამოიყენოს " webSocketDebuggerUrl` ქონება, როგორც
URL დაკავშირება პირდაპირ ინსპექტორი.

მოსმენის გამართვის შეტყობინებები სასიგნალო იგი `SIGUSR1` (Linux და
OS X). როგორც კვანძის 7 ეს ააქტიურებს მემკვიდრეობა Debugger API; კვანძის 8 და შემდეგ
იგი ააქტივებს ინსპექტორი API.

## უსაფრთხოების მნიშვნელობა

მას შემდეგ, რაც debugger აქვს სრული წვდომა Node.js შესრულების გარემოს, 
მუქარის მსახიობი შეუძლია დაკავშირება ამ პორტი შეუძლია შეასრულოს უკანონო
კოდი სახელით კვანძის პროცესი. ეს არის მნიშვნელოვანი, რომ გავიგოთ, უსაფრთხოების
შედეგების გამოვლენა debugger პორტი საჯარო და კერძო ქსელები.

### გამოაშკარავებას გამართვის პორტი საჯაროდ არის სახიფათო

თუ debugger არის ვალდებული საჯარო IP მისამართი, ან 0.0.0.0, ნებისმიერი კლიენტებს, რომ
შეგიძლიათ მიღწევა თქვენი IP მისამართი შეძლებთ დაუკავშირდეთ debugger გარეშე
შეზღუდვა და შეძლებს აწარმოებს თვითნებური კოდი.

By default `კვანძის --შეამოწმოს` ავალდებულებს 127.0.0.1. თქვენ მკაფიოდ უნდა უზრუნველყოს
საჯარო IP მისამართი ან 0.0.0.0 და ა. შ., თუ თქვენ აპირებთ უნდა დაუშვას გარე კავშირები
რომ debugger. ამით შეიძლება გამოამჟღავნონ თქვენ პოტენციურად მნიშვნელოვანი უსაფრთხოების
საფრთხე. ჩვენ გთავაზობთ უზრუნველყოს შესაბამისი ეკრანები და დაშვების კონტროლის ადგილი
თავიდან ასაცილებლად უსაფრთხოების გამოფენა.

იხილეთ განყოფილება " [საშუალებას დისტანციური გამართვის სცენარი](#საშუალებას-დისტანციური გამართვის-სცენარი)' რამდენიმე რჩევა, თუ როგორ
უსაფრთხოდ საშუალებას დისტანციური დიაგნოსტიკა კლიენტებს დაკავშირება.

### ადგილობრივი პროგრამები აქვთ სრული წვდომა ინსპექტორი

მაშინაც კი, თუ თქვენ სავალდებულოა ინსპექტორი პორტი 127.0.0.1 (ნაგულისხმები), ნებისმიერი პროგრამები
გაშვებული ლოკალურად თქვენს კომპიუტერს ექნება დაუბრკოლებლად. ეს არის დიზაინი
საშუალებას ადგილობრივი debuggers შეძლებს ანიჭებენ მოხერხებულად.

### ბრაუზერები, WebSockets და იმავე წარმოშობის პოლიტიკა

საიტები ღია ვებ-ბრაუზერის შეუძლია WebSocket და HTTP მოთხოვნების ქვეშ
ბროუზერის უსაფრთხოების მოდელი. საწყის HTTP კავშირი აუცილებელია, მიიღოს
უნიკალური debugger სხდომა პირადობის მოწმობა. იგივე-წარმოშობა-პოლიტიკა ხელს უშლის საიტებზე მიმდინარეობს
შეუძლია ეს HTTP კავშირი. დამატებითი უსაფრთხოების წინააღმდეგ
[DNS rebinding თავდასხმები](https://en.wikipedia.org/wiki/DNS_rebinding), Node.js
ადასტურებს, რომ "მასპინძელი', headers კავშირი ან
მიუთითეთ IP მისამართი ან `localhost` ან `localhost6` ზუსტად.

ეს უსაფრთხოების პოლიტიკის აუკრძალეთ დამაკავშირებელი დისტანციური გამართვის სერვერზე
სასურველი hostname. თქვენ შეგიძლიათ მუშაობა-დაახლოებით ამ შეზღუდვის მითითებით
ან IP მისამართი ან გამოყენებით ssh გვირაბები, როგორც აღწერილია ქვემოთ.

## ინსპექტორი კლიენტებს

რამდენიმე კომერციულ და ღია ინსტრუმენტები შეიძლება დაკავშირება Node-ს ინსპექტორი. ძირითადი
ინფორმაცია ამ შემდეგნაირად:

#### [node-შეამოწმოს](https://github.com/nodejs/node-inspect)

* CLI Debugger მხარდაჭერით Node.js ფონდი, რომელიც იყენებს [ინსპექტორი ოქმი][].
* ვერსია არის შეფუთული კვანძის და შეიძლება გამოყენებულ იქნას `კვანძის შეამოწმოს myscript.js`.
* უახლესი ვერსია ასევე შეიძლება დამონტაჟებული დამოუკიდებლად (მაგალითად, `npm ინსტალაცია -გ node-შეამოწმოს`)
და მეორადი " node-შეამოწმოს myscript.js`.

#### [Chrome DevTools](https://github.com/ChromeDevTools/devtools-frontend) 55+

* **ვარიანტი 1**: Open `chrome://შეამოწმოს " ქრომის დაფუძნებული
ბრაუზერში. დააჭირეთ კონფიგურაცია ღილაკს და დარწმუნდით, რომ თქვენი სამიზნე მასპინძელი და პორტი
ჩამოთვლილი.
* **ვარიანტი 2**: დააკოპირეთ `devtoolsFrontendUrl "გამომავალი" /json/list`
(იხილეთ ზემოთ) ან --შეამოწმოს მინიშნება ტექსტი და პასტა შევიდა Chrome.
* **ვარიანტი 3**: დააყენოთ Chrome გაფართოება NIM (კვანძის ინსპექტორი მენეჯერი): 
https://chrome.google.com/webstore/detail/nim-node-inspector-manage/gnhhdgbaldcilmgcpfddgdbkhjohddkj

#### [Visual Studio კოდი](https://github.com/microsoft/vscode) 1.10+

* გამართვის პანელი, დააჭირეთ პარამეტრები ხატი, გახსნა".vscode/launch.json`.
აირჩიეთ "Node.js" პირველადი კონფიგურაცია.

#### [Visual Studio](https://github.com/Microsoft/nodejstools) 2017

* აირჩიეთ "გამართვის > დაწყება გამართვის" მენიუდან ან მოხვდა F5.
* [დეტალური ინსტრუქციები](https://github.com/Microsoft/nodejstools/wiki/Debugging).

#### [JetBrains WebStorm](https://www.jetbrains.com/webstorm/) 2017.1+ და სხვა JetBrains IDEs

* შექმენით ახალი Node.js გამართვის კონფიგურაცია და მოხვდა გამართვის. `--შეამოწმოს` გამოყენებული იქნება
 მიერ ნაგულისხმევი Node.js 7+. გამორთოთ მონიშვნის მოხსნა `js.debugger.კვანძი.გამოყენება.შეამოწმოს"
IDE რეესტრში.

#### [chrome-remote-ინტერფეისი](https://github.com/cyrus-and/chrome-remote-interface)

* ბიბლიოთეკა განმუხტვის კავშირების ინსპექტორი ოქმი წერტილი.

## საშუალებას დისტანციური გამართვის სცენარი

ჩვენ გირჩევთ, რომ თქვენ არასოდეს არ debugger მოუსმინოს საჯარო IP მისამართი. თუ
თქვენ უნდა საშუალებას დისტანციური გამართვის კავშირები ჩვენ გირჩევთ გამოიყენოთ ssh
გვირაბების ნაცვლად. ჩვენ გთავაზობთ შემდეგი მაგალითი საილუსტრაციო მიზნებისთვის.
გთხოვთ მესმის უსაფრთხოების რისკი, რომელიც საშუალებას დისტანციური წვდომის პრივილეგირებული
მომსახურების გაგრძელებამდე.

ვთქვათ, თქვენ გაშვებული კვანძის დისტანციური მანქანა, remote.example.com,, რომ თქვენ
მინდა, რომ შეძლებს გამართვის. რომ მანქანა, თქვენ უნდა დაიწყოს კვანძის პროცესი
ერთად ინსპექტორი მოსმენის მხოლოდ localhost (default).

``bash
$ კვანძის --შეამოწმოს server.js
``

ახლა, თქვენი ადგილობრივი მანქანა, სადაც გსურთ, რომ დაიწყოს გამართვის კლიენტი
დაკავშირებით, თქვენ შეგიძლიათ setup ssh გვირაბი:

``bash
$ ssh -L 9221:localhost:9229 user@remote.example.com
``

ამ იწყება ssh გვირაბი სხდომა, სადაც კავშირი პორტი 9221 თქვენი ადგილობრივი
მანქანა იქნება გადაგზავნილი პორტი 9229 შესახებ remote.example.com. ახლა თქვენ შეგიძლიათ ანიჭებენ
a debugger, როგორიცაა Chrome DevTools ან Visual Studio კოდი localhost:9221,
რომელიც უნდა შეეძლოს გამართვის თითქოს Node.js განცხადება იქნა გაშვებული ადგილობრივად.

## Legacy Debugger

**მემკვიდრეობა debugger უკვე მოძველებულია როგორც კვანძის 7.7.0. გთხოვთ, გამოიყენოთ --შეამოწმოს
და ინსპექტორი ნაცვლად.**

როდესაც დაიწყო **--გამართვის** ან **--გამართვის-brk** კონცენტრატორები ვერსია 7 და
ადრე, Node.js უსმენს გამართვის ბრძანებები განსაზღვრული შეწყვიტა
V8 გამართვის პროტოკოლი TCP პორტი, by default `5858`. ნებისმიერი debugger კლიენტი
რომელიც საუბრობს ამ ოქმის შეიძლება დაკავშირება და გამართვის გაშვებული პროცესი; 
წყვილი პოპულარული პირობა ჩამოთვლილია ქვემოთ.

V8 გამართვის ოქმი აღარ სასარგებლოდ ან დოკუმენტირებული.

#### [ინტეგრირებული Debugger](https://nodejs.org/dist/latest-v6.x/docs/api/debugger.html)

დაწყება "კვანძის გამართვის script_name.js" დაიწყოს თქვენი script ქვეშ კვანძის builtin
command-line debugger. თქვენი script იწყება კიდევ ერთი კვანძის პროცესი დაიწყო
"--გამართვის-brk "ვარიანტი, და საწყის კვანძის პროცესს გადის" _debugger.js`
სცენარი და აკავშირებს თქვენს სამიზნე.

#### [node-ინსპექტორი](https://github.com/node-inspector/node-inspector)

გამართვის თქვენი Node.js app Chrome DevTools გამოყენებით შუამავალი პროცესი
რომელიც ითარგმნება, ინსპექტორი ოქმი გამოიყენება ქრომის, რათა V8 Debugger
ოქმი გამოიყენება Node.js.
