# Sicherheitsberichterstattung

Für weitere Details zu aktiven Sicherheitsrichtlinien, checke [diese Seite](https://github.com/nodejs/node/security/policy).

## Fehler in Node.js melden

Melde Sicherheitsfehler in Node.js über [HackerOne](https://hackerone.com/nodejs).

Ihr Bericht wird innerhalb von 5 Tagen bestätigt und Sie erhalten innerhalb von 10 Tagen eine detailliertere Antwort auf Ihren Bericht
, die die nächsten Schritte in
anzeigt, wie Ihre Einreichung behandelt wird.

Nach der ersten Antwort auf Ihren Bericht das Sicherheitsteam wird versuchen,
über die Fortschritte auf dem Weg zu einer fixen und vollständigen Ankündigung auf dem Laufenden zu halten
und können weitere Informationen oder Hinweise zu dem gemeldeten
Problem anfordern.

### Node.js Bug Bounty Programm

Das Node.js Projekt beschäftigt sich mit einem offiziellen Bug Bounty Programm für
Forscher und verantwortliche öffentliche Enthüllungen. Das Programm wird über
die HackerOne Plattform verwaltet. Weitere Details findest du unter [https://hackerone.com/nodejs](https://hackerone.com/nodejs).

## Fehler in einem Drittanbieter-Modul melden

Sicherheitsfehler in Modulen von Drittanbietern sollten ihren jeweiligen
Betreuern gemeldet werden.

## Offenlegungsrichtlinie

Hier ist die Sicherheits-Offenlegungsrichtlinie für Node.js

- The security report is received and is assigned a primary handler. This
  person will coordinate the fix and release process. The problem is confirmed
  and a list of all affected versions is determined. Code is audited to find
  any potential similar problems. Fixes are prepared for all releases which are
  still under maintenance. These fixes are not committed to the public
  repository but rather held locally pending the announcement.

- Ein empfohlenes Embargodatum für diese Verwundbarkeit wird gewählt und ein CVE (Common
  Schwachstellen und Exposures (CVE®️)) wird für die Verwundbarkeit angefordert.

- Am Embargo-Datum wird der Sicherheits-Mailingliste von Node.js eine Kopie der
  Ankündigung gesendet. Die Änderungen werden in das öffentliche Repository gepresst und neue Builds
  werden in nodejs eingesetzt. Innerhalb von 6 Stunden nachdem die Mailingliste
  benachrichtigt wurde, wird eine Kopie der Ankündigungen im Node.js Blog veröffentlicht.

- Normalerweise wird das Embargodatum 72 Stunden seit der Ausstellung des Lebenslauf
  festgelegt. Dies kann jedoch abhängig von der Schwere des Fehlers oder von
  Schwierigkeiten bei der Anwendung eines Fehlers variieren.

- This process can take some time, especially when coordination is required
  with maintainers of other projects. Every effort will be made to handle the
  bug in as timely a manner as possible; however, it's important that we follow
  the release process above to ensure that the disclosure is handled in a
  consistent manner.

Die Open Source Security Foundation (OpenSSF) \[Best Practices badge]\(https://github. om/coreinfrastructure/best-practices-badge) ist ein Weg für Projekte von Free/Libre und Open Source Software (FLOSS) zu zeigen, dass sie den besten Praktiken folgen. Projekte können sich freiwillig selbst bescheinigen, wie sie jeder bewährten Praxis folgen. Die Verbraucher des Ausweises können schnell beurteilen, welche FLOSS-Projekte den besten Praktiken folgen und als Folge davon eher sichere Software von höherer Qualität produzieren.
