# Informes de seguridad

Para más detalles sobre las Políticas de Seguridad activas, revisa [esta página](https://github.com/nodejs/node/security/policy).

## Reportando un error en Node.js

Informar errores de seguridad en Node.js a través de [HackerOne](https://hackerone.com/nodejs).

Su informe será reconocido en 5 días, y recibirás una respuesta más
detallada a tu informe en un plazo de 10 días indicando los siguientes pasos en
para gestionar tu envío.

Después de la respuesta inicial a tu informe, el equipo de seguridad se esforzará por mantenerte informado del progreso hacia una solución y un anuncio completo, y puede solicitar información adicional o orientación sobre el problema reportado.

### Programa de recompensas de errores Node.js

El proyecto Node.js se involucra en un programa oficial de recompensas de errores para investigadores
de seguridad y divulgaciones públicas responsables. El programa se administra a través de
la plataforma HackerOne. Para más información, consulta [https://hackerone.com/nodejs](https://hackerone.com/nodejs).

## Reportando un error en un módulo de terceros

Los errores de seguridad en módulos de terceros deben ser reportados a sus respectivos mantenedores
.

## Política de divulgación

Aquí está la política de divulgación de seguridad para Node.js

- El informe de seguridad es recibido y se asigna un manejador principal. Esta persona coordinará el proceso de solución y lanzamiento. El problema es confirmado y se determina una lista de todas las versiones afectadas. Se audita el código para encontrar cualquier problema similar potencial. Se preparan soluciones para todas las versiones que aún están bajo mantenimiento. Estas soluciones no se confirman en el repositorio público, sino que se mantienen localmente a la espera del anuncio.

- Se ha elegido una fecha de embargo sugerida para esta vulnerabilidad y se solicita una CVE (Common
  Vulnerabilities and Exposures (CVE®️)) para la vulnerabilidad.

- En la fecha del embargo, la lista de correo de seguridad de Node.js se envía una copia del anuncio
  . Los cambios se colocan en el repositorio público y las nuevas compilaciones
  se implementan en nodejs. rg. Dentro de las 6 horas posteriores a que la lista de correo sea
  notificada, se publicará una copia del aviso en el blog de Node.js.

- Normalmente la fecha del embargo se establecerá 72 horas a partir de la hora en que el CVE sea
  emitido. Sin embargo, esto puede variar dependiendo de la gravedad del error o de la dificultad
  en la aplicación de una solución.

- Este proceso puede tardar algún tiempo, especialmente cuando se requiere coordinación
  con los mantenedores de otros proyectos. Se harán todos los esfuerzos para manejar el
  bug de la manera más oportuna posible; sin embargo, es importante que sigamos
  el proceso de liberación anterior para asegurar que la divulgación se maneje de una manera
  consistente.

## Recibiendo actualizaciones de seguridad

Las notificaciones de seguridad se distribuirán a través de los siguientes métodos.

- [Grupo de Google](https://groups.google.com/group/nodejs-sec)
- [Blog de Node.js](/blog)

## Comentarios sobre esta política

Si tienes sugerencias sobre cómo mejorar este proceso, por favor envía un
[pull request](https://github.com/nodejs/nodejs.org) o
[file an issue](https://github.com/nodejs/security-wg/issues/new) para discutir.

The Open Source Security Foundation (OpenSSF) \[La insignia de mejores prácticas]\(https://github. om/coreinfrastructure / best practices-badge) es una manera para los proyectos de Software Libre y Libre (FLOSS) de mostrar que siguen las mejores prácticas. Los proyectos pueden certificar voluntariamente cómo siguen cada una de las mejores prácticas. Los consumidores del distintivo pueden evaluar rápidamente qué proyectos FLOSS están siguiendo las mejores prácticas y, como resultado, tienen más probabilidades de producir software seguro de mayor calidad.
